# MCP Tools for Claude Code — Comprehensive Idea List

Tools are grouped by domain. Each entry: tool name, what it does, why it helps Claude Code.

================================================================
1. CODE INTELLIGENCE & NAVIGATION
================================================================
- symbol_lookup        Resolve a symbol to its definition, type, and docstring across the repo (LSP-backed). Saves grep churn.
- find_references       All usages of a symbol with file:line. Better than text grep (scope-aware).
- call_hierarchy        Callers/callees of a function. Helps impact analysis before edits.
- type_at_point         Hover type info at a file:line. Removes guessing about inferred types.
- rename_symbol         Project-wide safe rename via LSP. Avoids broken text replacements.
- import_resolver       Given a symbol, suggest the correct import path/module.
- dead_code_finder      Unused exports, functions, vars. Cleanup targeting.
- dependency_graph      Module import graph (in/out edges) for a file or package.
- ast_query             Run a structural query (tree-sitter/ast-grep) and return matches. Precise refactors.
- code_outline          Return file structure (classes/functions/signatures) without full file read. Token saver.

================================================================
2. SEARCH & RETRIEVAL (token-efficient)
================================================================
- semantic_search       Embedding search over the codebase; return ranked snippets for a natural-language query.
- definition_grep       Grep that returns only definitions (def/class/fn), not every mention.
- recent_changes_search Search only files changed in last N commits/days.
- doc_search            Search local docs/README/wiki and return relevant sections.
- example_finder        Find existing usage examples of an API/pattern in the repo.
- regex_extract         Return only capture groups, not whole lines. Compact output.

================================================================
3. GIT & VERSION CONTROL
================================================================
- git_blame_range       Blame for a line range with commit messages (why was this written).
- commit_context        For a commit, return diff summary + PR link + related issues.
- branch_diff_summary    Semantic summary of what differs between two branches.
- find_introducing_commit  Bisect-style: which commit introduced a string/bug.
- staged_diff           Return staged diff only, compacted.
- conflict_explainer    Parse merge conflicts and explain both sides + suggest resolution.
- changelog_since       Generate changelog from commits since a tag.
- pr_metadata           Fetch PR title/body/reviews/checks for current branch.

================================================================
4. BUILD, TEST & DIAGNOSTICS
================================================================
- run_tests_filtered    Run tests matching a pattern; return only failures + stack frames in repo.
- test_for_file         Map a source file to its test file(s) and vice versa.
- coverage_gaps         Lines/branches in a file not covered by tests.
- build_errors          Run build, return errors grouped by file/rule (compact).
- flaky_test_detector   Run a test N times, report nondeterministic ones.
- typecheck             Run type checker, return errors grouped by code.
- lint_fixable          List lint issues + which are auto-fixable.
- benchmark_run         Run a microbenchmark and return before/after timings.

================================================================
5. RUNTIME / EXECUTION SANDBOX
================================================================
- run_snippet           Execute a code snippet in an isolated sandbox, return stdout/stderr/exit.
- repl_session          Persistent REPL (python/node) across calls for iterative debugging.
- http_probe            Make an HTTP request to a local/dev endpoint and return compact response.
- db_query              Run a read-only query against a dev DB, return rows (truncated).
- schema_introspect     Return DB schema (tables/columns/indexes/relations) without values.
- env_inspector         List env vars (redacted) and config resolution for the project.
- process_inspector     What's listening on which ports / running dev servers.

================================================================
6. DOCUMENTATION & KNOWLEDGE
================================================================
- library_docs          Fetch official docs for a specific library@version + symbol (like Context7).
- api_signature         Resolve a third-party function's signature/params from installed package.
- changelog_lookup      Fetch a dependency's changelog between two versions (upgrade planning).
- adr_search            Search architecture decision records for rationale.
- internal_wiki         Query Confluence/Notion/internal docs.
- code_comments_index   Surface TODO/FIXME/HACK with file:line and age.

================================================================
7. DEPENDENCY & SUPPLY CHAIN
================================================================
- dep_tree              Resolved dependency tree (compact) with versions.
- outdated_deps         Outdated packages + latest + semver risk.
- vuln_scan             Known CVEs in dependencies (advisory DB).
- license_audit         License of each dependency + flag incompatibilities.
- why_installed         Why a transitive dep is present (which parent pulls it).
- bundle_size           Size impact of a package/import.

================================================================
8. PROJECT MEMORY & CONTEXT
================================================================
- project_facts         Store/retrieve durable facts about the project (build cmds, conventions).
- task_journal          Append/read a running log of what was changed this session.
- convention_lookup     Return coding conventions/patterns inferred from the repo.
- glossary              Domain term definitions specific to this codebase.
- decision_log          Record why a particular approach was chosen (avoids re-litigating).

================================================================
9. ISSUE TRACKING & COLLABORATION
================================================================

- linear_issue          Same for Linear.
- github_issue          Read/triage/comment on GitHub issues.
- slack_post            Post a status/notification to a channel (with approval).

- standup_summary       Generate a standup update from today's commits/PRs.

================================================================
10. CI/CD & DEPLOYMENT
================================================================
- ci_status             Latest CI run status + only failing job logs (compact).
- ci_logs_failed        Fetch just the failing step's logs, filtered to errors.
- deploy_status         Current deploy state per environment.

================================================================
11. OBSERVABILITY & DEBUGGING
================================================================
- log_query             Query app logs (Loki/CloudWatch/Datadog) filtered + deduped with counts.
- error_groups          Top error groups from Sentry with stack + occurrence count.
- trace_lookup          Fetch a distributed trace by ID, return span tree.
- metric_query          Query a metric (Prometheus) and return summary stats.
- stack_decode          Symbolicate/decode a minified or native stack trace.
- repro_capture         Capture a failing request/payload as a reproducible test case.

12. REFACTORING & CODE QUALITY
================================================================

- extract_function      AST-safe extract of a selection into a function.
- codemod_apply         Apply a structural transform across files (ast-grep/jscodeshift).
- format_check          Which files violate formatting (no diffs, just list).
- naming_consistency    Flag inconsistent naming for similar concepts.

================================================================
13. SECURITY (defensive)
================================================================
- secret_scan           Detect committed secrets/keys with file:line (redacted).
- sast_scan             Static analysis for injection/auth/crypto issues.
- dependency_audit      (see vuln_scan) integrated security advisories.
- iac_scan              Misconfigurations in Terraform/K8s/Dockerfiles.
- permission_diff       What perms/scopes a change adds (IAM, OAuth).
- auth_flow_map         Map where authn/authz checks exist for an endpoint.

================================================================
14. DATA & API CONTRACTS
================================================================
- openapi_lookup        Resolve an endpoint's request/response schema from OpenAPI spec.
- graphql_introspect    Return GraphQL schema types/fields for a query.
- proto_lookup          Resolve protobuf/gRPC message definitions.
- mock_generator        Generate realistic mock data conforming to a schema.
- contract_diff         Detect breaking API changes between two spec versions.
- json_shape            Return JSON structure/types without bulky values.

15. FILE & WORKSPACE OPS
================================================================
- file_tree             Compact tree of a directory respecting .gitignore.
- large_file_summary    Summarize a huge file's structure instead of dumping it.
- diff_apply_safe       Apply a patch with conflict/hunk validation.
- scratch_workspace     Create/manage a temp worktree for risky experiments.
- file_history          Show how a file evolved (key revisions).
- binary_inspect        Metadata for binary/asset files (dimensions, size, type).

16. MULTIMODAL / DESIGN

17. AGENTIC / WORKFLOW HELPERS
================================================================

- impact_estimator      Estimate blast radius of a proposed change (files/tests affected).
- checkpoint_create     Snapshot working state (stash/commit) before a risky step.
- approval_gate         Pause and request human approval for a destructive action.
- cost_tracker          Track token/compute spend for the session.
- session_handoff       Serialize current context into a resumable handoff doc.

================================================================
18. DOMAIN-SPECIFIC (pick per stack)
================================================================
- react_component_map   Component tree + props flow for a React app.
- rust_macro_expand     Expand a Rust macro to see generated code.
- sql_explain           EXPLAIN ANALYZE a query and summarize the plan.
- docker_inspect        Compact container/image/layer info.
- k8s_resource          Get/describe a k8s resource, filtered.
- terraform_plan_summary Summarize a `terraform plan` (adds/changes/destroys).

================================================================
DESIGN PRINCIPLES FOR THESE TOOLS
================================================================
- Token-efficient by default: return structured, filtered output, not raw dumps.
- Read-only unless explicitly mutating; mutating tools require approval gates.
- Scope-aware (LSP/AST) beats text grep — fewer false positives.
- Composable: small single-purpose tools Claude can chain.
- Idempotent + safe-to-retry where possible.
- Include file:line references so output is clickable/actionable.
- Prefer summaries with drill-down (return IDs/handles to fetch detail on demand).


Here's a broad set of ideas across different domains, from practical to creative:

---

## Developer productivity

- **Code reviewer** — analyses a diff or file for code smells, security issues, naming conventions, and suggests improvements
- **Git history analyser** — summarises what changed in a repo over a time range, who changed what, identifies hotspot files
- **Dependency auditor** — scans a `pom.xml` or `package.json` and reports outdated, vulnerable, or unused dependencies
- **Test coverage advisor** — reads a class and its test file, identifies untested paths and suggests missing test cases
- **OpenAPI validator** — validates a spec file, checks for missing descriptions, inconsistent naming, broken `$ref` links
- **SQL query explainer** — takes a SQL query + schema and explains the execution plan, suggests indexes
- **Log pattern detector** — reads log files and surfaces recurring error patterns, anomalies, or performance spikes
- **Tech debt tracker** — scans source files for `TODO`, `FIXME`, `HACK` comments and produces a structured report

---

## Financial / Yapily-specific 🏦

- **Transaction categoriser** — takes raw transaction data and classifies each into a category (groceries, travel, utilities etc)
- **Spending anomaly detector** — compares current month vs historical average and flags unusual spend
- **Account health checker** — given account data, produces a plain-English summary of financial health
- **Recurring payment finder** — identifies likely subscriptions and direct debits from transaction history
- **Balance forecaster** — projects future balance based on historical patterns and known upcoming payments
- **Consent status checker** — looks up the status of an open banking consent and explains it in plain English
- **Institution availability monitor** — checks if a bank's API is responding and reports degraded connections
- **Payment risk scorer** — given a payment request, scores it for risk based on amount, recipient, account history

---

## Data & documents

- **PDF summariser** — takes a PDF path or URL, extracts and summarises the key points
- **CSV profiler** — reads a CSV and reports column types, nulls, distributions, outliers
- **Schema diff tool** — compares two database schemas (or JSON schemas) and describes what changed
- **Data quality checker** — validates a dataset against rules (no nulls in required fields, values in range, etc)
- **Markdown to report converter** — turns structured markdown into a formatted HTML or PDF report
- **Contract clause extractor** — reads a legal document and extracts key clauses (termination, liability, payment terms)

---

## Infrastructure & ops

- **Docker inspector** — lists running containers, their resource usage, and flags anything unhealthy
- **Environment diff tool** — compares environment variables across two profiles (dev vs prod) and highlights gaps
- **Dead endpoint detector** — reads an OpenAPI spec and checks which endpoints return 404 or error in a target environment
- **Config validator** — reads a Spring `application.properties` / `yaml` and validates required keys are present and correctly typed
- **Kubernetes pod summariser** — queries a cluster and gives a human-readable health summary
- **Deployment diff** — compares what's running in staging vs production

---

## Communication & workflow

- **Slack message drafter** — given a situation description, drafts a Slack message in the right tone
- **Meeting notes formatter** — takes raw bullet points and formats them into structured meeting minutes with action items
- **JIRA ticket writer** — given a bug description or feature idea, generates a well-structured ticket with AC
- **PR description generator** — given a diff summary, writes a clear pull request description
- **Runbook generator** — given a service description, generates an incident runbook template
- **On-call alert explainer** — takes a PagerDuty/alert payload and explains in plain English what fired and why

---

## AI & knowledge

- **Prompt optimiser** — takes a raw prompt and rewrites it to be clearer and more effective
- **Knowledge base searcher** — queries an internal Confluence/Notion space and returns relevant pages
- **Decision logger** — records architectural or product decisions with context, date, and rationale to a store
- **Glossary builder** — scans a codebase or document set and builds a domain glossary from recurring terms
- **Q&A over docs** — given a folder of markdown files, answers questions using their content

---

## Fun / creative

- **Standup generator** — given your git commits from yesterday, writes your daily standup update
- **Code namer** — suggests good names for a class, method, or variable given what it does
- **Retro facilitator** — given a list of events from the sprint, suggests what went well / what to improve
- **Complexity scorer** — rates code complexity and gives it a human-readable label ("a junior could follow this" → "only the author understands this")
- **Emoji commit tagger** — reads a commit message and suggests the right gitmoji prefix

---

The best ones to build first are usually those that **save you or your team time today** — the git history analyser, PR description generator, or log pattern detector are all things you'd use constantly inside Claude Code itself.

* `go_to_definition` — jump from a symbol to its declaration.
* `find_references` — find all usages of a symbol.
* `outline_file` — return the structural outline of a file.

* `summarize_directory` — summarize a package/module/folder.
* `dependency_graph` — show import/module/package dependencies.
* `call_graph` — show function/method call relationships.
* `type_hierarchy` — show inheritance / interface implementation trees.
* `detect_frameworks` — infer frameworks, runtimes, and build systems in use.
* `detect_languages` — identify languages and versions in the repo.
* `list_entrypoints` — find app entrypoints, main binaries, handlers, CLI commands.

## A strong design rule

For Claude Code, the best MCP tools are usually:

* **small and atomic** rather than giant “do everything” endpoints,
* **typed and structured** rather than free-form text blobs,
* **read-optimized** before write-optimized,
* and **safe by default** with dry-run, preview, and confirmation modes.

* `apply_patch_preview` → `apply_patch`

Building an MCP (Model Context Protocol) server for Claude Code is a fantastic way to extend its capabilities. Because Claude Code natively handles basic file editing, `grep` searches, and standard shell command execution, the most valuable MCP tools are those that bridge the gap to **structured data, complex external systems, and stateful environments** without overwhelming the token context window.

Here is a comprehensive list of high-leverage MCP tools you could build, categorized by the problems they solve.

## 1. Database & State Introspection

Claude Code can technically run `psql` or `redis-cli` via bash, but the raw text output is token-heavy and prone to formatting errors. An MCP server provides structured, context-efficient access.

* **Safe Schema & Query Engine (PostgreSQL/SQL):**
* **Tools:** `get_schema_diagram`, `execute_readonly_query`, `explain_query_plan`.
* **Why it's useful:** Allows Claude to intelligently explore database structures and test queries without the risk of accidentally running an `UPDATE` or `DROP` command via a raw bash shell.


* **Key-Value & Cache Inspector (Redis):**
* **Tools:** `scan_keys_by_pattern`, `get_decoded_value`, `monitor_pubsub_channel`.
* **Why it's useful:** Perfect for debugging asynchronous message queues, orchestrators, or caching layers. Instead of dumping raw bytes, the MCP can decode JSON/MessagePack payloads into clean context.


* **Time-Series & Telemetry Viewer:**
* **Tools:** `fetch_metric_range`, `detect_anomalies`, `summarize_logs`.
* **Why it's useful:** For analyzing transaction spikes or system load. The MCP can pre-aggregate massive time-series datasets into statistical summaries, saving hundreds of thousands of tokens.



## 2. API & Network Engineering

When working with complex external APIs (like payment gateways, open banking protocols, or industrial endpoints), Claude needs more than just `curl`.

* **OpenAPI/Swagger Sandbox:**
* **Tools:** `load_openapi_spec`, `generate_mock_payload`, `execute_api_call`.
* **Why it's useful:** Instead of pasting API docs into the chat, the MCP dynamically parses local or remote swagger docs, allowing Claude to understand endpoints, construct valid JSON payloads, and test them instantly.


* **Local Webhook Catcher:**
* **Tools:** `start_webhook_listener`, `get_latest_payloads`, `clear_webhook_history`.
* **Why it's useful:** If you are building integrations that rely on external triggers, this MCP can spin up a lightweight local server (or tunnel) and feed the incoming webhook payloads directly to Claude for debugging.


* **Synthetic Data Generator:**
* **Tools:** `generate_transaction_batch`, `mock_iot_telemetry`.
* **Why it's useful:** When writing parsing logic or orchestrators, you often need realistic dummy data. An MCP can dynamically generate thousands of lines of schema-compliant data (e.g., banking transactions or SCADA sensor data) and write them straight to a local test database.



## 3. Deep Code & Architecture Intelligence

Claude Code’s native file search is text-based. An AST-aware (Abstract Syntax Tree) tool makes it significantly smarter, especially for strongly typed languages like Rust or Java.

* **Semantic AST Navigator:**
* **Tools:** `find_trait_implementations`, `get_method_callers`, `extract_interface_signature`.
* **Why it's useful:** If you ask Claude to "find everything that implements this interface," a text search will fail if the implementation is in another file without the explicit keyword. An AST MCP uses tools like `rust-analyzer` or Java's LSP to return structural truths.


* **Architecture & Dependency Mapper:**
* **Tools:** `generate_module_graph`, `find_circular_dependencies`.
* **Why it's useful:** It can analyze a complex codebase and feed Claude a JSON representation of how modules communicate, allowing it to propose architectural refactors rather than just line-by-line edits.



## 4. Cross-Platform Environment & System Debugging

Managing different paths, dependencies, and binaries across macOS and Windows is notoriously frustrating.

* **Environment Validator:**
* **Tools:** `resolve_binary_path`, `diff_env_variables`, `check_port_usage`.
* **Why it's useful:** If a build script works on macOS but fails on Windows 10, Claude can use this MCP to systematically probe system paths, registry keys (on Windows), or dotfiles (on Mac) to find the discrepancy.


* **Smart Log Slicer:**
* **Tools:** `extract_stack_traces`, `filter_logs_by_level`, `tail_recent_errors`.
* **Why it's useful:** Native `cat` or `tail` commands easily blow out the context window. A smart log MCP parses standard log formats, filters out the noise, and only feeds Claude the actual stack traces or error blocks.



## 5. Domain-Specific Simulators

If you work in niche intersections—like connecting IT infrastructure to physical hardware—mocking that environment is crucial.

* **Hardware/OT Protocol Emulator:**
* **Tools:** `start_modbus_server`, `read_virtual_register`, `inject_hardware_fault`.
* **Why it's useful:** Allows Claude to write and test code meant for physical hardware by interacting with an MCP server that accurately mimics serial ports, PLCs, or flight telemetry data streams.
